#### 아이템 7: 타입을 값의 집합으로 보기

타입을 이름이나 속성 목록이 아니라 **할당 가능한 값의 집합**으로 생각하니 구조적 타이핑을 더 자연스럽게 이해할 수 있었다.
타입스크립트가 타입의 동일성보다 “이 값을 저 타입에 할당할 수 있는가?”를 자주 검사하는 이유도 집합의 포함 관계로 연결됐다.
`unknown`이 모든 JavaScript 값을 포함하는 최상위 타입이라는 사실도 집합 관점에서는 직관적이었다.
아이템 4에서 배운 “객체 타입은 정확히 그 속성만 가진 값이 아니다”라는 내용과도 이어졌다.
구조적 타이핑에서 객체 타입은 필요한 속성을 가진 값들의 집합이다.
따라서 필요한 속성 외에 다른 속성을 가진 값도 같은 집합에 포함될 수 있다.

### 기존 직관이 뒤집힌 지점

#### 인터섹션은 왜 공통 속성만 남기지 않을까?

처음에는 교집합이므로 두 객체 타입의 공통 속성만 남을 것이라고 생각했다.
하지만 인터섹션은 **속성 이름의 교집합**이 아니라 **두 타입에 모두 할당 가능한 값의 집합**이다.
따라서 객체 값이 두 타입을 동시에 만족하려면 일반적으로 양쪽이 요구하는 속성을 모두 가져야 한다.

#### `extends`로 필드를 추가했는데 왜 값의 집합은 좁아질까?

인터페이스의 필드는 늘어나지만, 모든 필드를 만족해야 하므로 실제로 할당 가능한 값의 수는 줄어든다.
상속 관점에서는 타입을 “확장”하지만, 집합 관점에서는 기존 타입의 **부분집합**을 만드는 셈이다.
속성의 개수와 값 집합의 크기가 반대로 움직일 수 있다는 점이 새로웠다.

#### 타입 검사는 언제 일어날까?

언어 서비스는 코드를 작성하는 동안 할당 가능성을 계속 검사한다.
`tsc`도 컴파일 과정에서 타입을 검사하고 TypeScript를 JavaScript로 변환한다.
실제 런타임에는 타입 정보가 사라지고 변환된 JavaScript만 브라우저나 Node.js에서 실행된다.

### 아직 쉽게 이해되지 않는 부분

#### 구조적 타이핑과 잉여 속성 체크의 관계

객체 리터럴을 직접 전달할 때 추가 속성 때문에 오류가 나면 구조적 타이핑이 적용되지 않는 것처럼 느껴졌다.
현재의 이해로는 구조적 타이핑이 꺼지는 것이 아니라, 새 객체 리터럴의 오타를 잡기 위한 **추가 검사**가 함께 적용되는 것이다.
같은 객체를 변수에 먼저 담으면 통과할 수 있다는 차이가 여전히 낯설어서 예제를 더 만들어볼 필요가 있다.

#### 집합 관점과 불변성의 관계

타입의 값 영역이 비슷해 보여도 제네릭이나 가변성 조건 때문에 할당 가능성이 달라질 수 있다는 점은 아직 명확하지 않다.
공변성, 반공변성, 불변성이 집합의 포함 관계를 어떻게 바꾸는지 별도로 확인하고 싶다.

### 아이템9: 타입 단언 대신 타입 선언 사용하기

타입 선언 > 타입 단언
그렇지만 타입 단언이 **반드시** 필요한 경우도 있네?
DOM을 조작하는 경우: TS는 DOM에 접근할 수 없구나!
문맥상 null이 될 가능성이 없다는 걸 안다면, 단언으로 null을 제외할 수 있음
null일 가능성이 있다면 옵셔널 체이닝(?) 사용을 편리하다.
(!)연산자와 비슷해 보이지만 다른 의미를 가진다.
옵셔널 체이닝 연산자는 **JS의 문법**이다!
단언이 만능은 아니다. 서로의 서브타입이 아닌 경우는 변환이 불가능하다.
이걸 해결하려면 unknown 사용, but 매우 찝찝.. 코드 위험...
`as const`: 타.입 단언이라기 보다는 타입을 더 구체적이고 안전하게 만드는 구문
어떨 때 사용하더라? -> 상수라고 주장하는 것

1. 단순히 리터럴을 사용하고 싶을 때
2. 객체를 읽기 전용으로 만들 때
3. 배열을 읽기 전용의 튜플로 만들 때
   타입을 명시하지 않는 경우

```ts
function createColourVariant(colour: Colour, variant: Variant) {
  return `${variant}-${colour}`;
}
```

TS는 반환값을 `string`으로 추론한다.
타입을 명시한 경우

```ts
type ColourVariant = `${Variant}-${Colour}`;
// "light-red" | "light-green" | "light-blue" | "dark-red" | ...

function createColourVariant(colour: Colour, variant: Variant): ColourVariant {
  return `${variant}-${colour}`;
}
```

`ColourVariant`라는 새 타입을 미리 정의해서, "가능한 조합 6개 중 하나"라고 명시적으로 알려줘요.
장점: 이 타입을 다른 곳에서도 재사용할 수 있음.
단점: 타입을 하나 더 만들어야 해서 코드가 늘어남.
`as const`로 단언한 경우

```ts
function createColourVariant(colour: Colour, variant: Variant) {
  return `${variant}-${colour}` as const;
}
```

`as const`는 "이 값을 넓은 `string`이 아니라, 가능한 한 가장 좁고 정확한 타입으로 취급해줘"라는 의미를 가진다.
그러면 TypeScript가 별도의 `type ColourVariant` 없이도 자동으로 `"light-red" | "light-green" | ... ` 같은 정확한 타입을 추론한다.
어떻게 이게 가능한 걸까?
템플릿 리터럴 타입 안에 유니온 타입이 존재하면, TypeScript는 가능한 모든 조합을 자동으로 곱해서 새로운 유니온 타입을 만든다.
이건 TS의 템플릿 리터럴 타입 시스템의 기능이다. (일반 JS 문자열 템플릿과 달리, 타입 레벨에서 조합을 계산해줌)

### 아이템10: 객체 래퍼 타입 피하기

우리가 기본형에 대해 사용하던 그 타입들(`String`, `Number`, ...)이 맞다.
기본형을 각자의 객체로 래핑해서 래퍼 타입이라고 하는구나!
JS의 프로토타입과 관련이 있을까?
기본형 타입은 객체 래퍼에 할당할 수 있다. 기본형 타입을 객체 래퍼에 할당하는 선언을 허용한다. <- 이 말이 어렵다. 쉽게 설명하면 뭘까?
기본형 타입을 객체 래퍼에 할당한다는 의미:

```ts
// 문제 없는 코드!

let str: string = 'hello';
let strObj: String = str;
```

반대로 객체 래퍼를 기본형에 할당한다는 의미:

```ts
// 오류!

let strObj: String = new String('hello');
let str: string = strObj;
```

왜 헷갈리기 쉽다는 말을 할까?

```ts
function greet(name: String) {
  // 실수로 대문자로 씀
  console.log(`Hello, ${name}`);
}

greet('Alice'); // 동작하긴 하는데..
```

()기본형 타입 -> 객체 래퍼) 할당이 가능하므로 대문자 `String`을 타입으로 써도 문법적으로 에러 X
근데 `String` 객체는 `.length`, `.toUpperCase()` 같은 메서드는 있어도, 원시값처럼 `===` 비교가 제대로 안 되거나 예상치 못한 동작을 할 수 있다.
결론은 그냥 소문자 써라.

### 아이템 12: 함수 표현식에 타입 적용하기

**함수의 매개변수부터 반환값까지 전체를 함수 타입으로 선언해 함수 표현식에 재사용할 수 있다는 장점이 있다 ← 이게 어떤 의미일까?**

```ts
// 1. 함수 문장
// a, b, 반환값의 타입을 계속 일일히 작성해줘야 한다.
function add(a: number, b: number): number {
  return a + b;
}

// 2. 함수 표현식
// 함수 자체를 타입으로 만들어서, 재사용이 가능하다.
type MathOp = (a: number, b: number) => number;

const add: MathOp = (a, b) => a + b;
const subtract: MathOp = (a, b) => a - b;
```

반복되는 함수 시그니처를 하나의 함수 타입으로 통합하는 데 의미가 있다. 재활용!

**라이브러리들은 공통 함수 시그니처를 타입으로 제공하기도 한다.**

fetch 타입이 있다는 사실은 처음 알았다! TS가 input과 init의 타입을 추론할 수 있게 한다. (fetch 메서드에 `typeof`를 붙여 타입으로 가져온다)

**매개변수는 기존 걸 사용하면서, 반환 타입만 변경하고 싶은 경우 `Parameters` 유틸리티 타입을 사용하자.**

반대로 반환 타입만 가져오고 싶다면 `ReturnType` 유틸리티 타입을 사용하자.

**`Parameters` 유틸리티 타입을 사용하는 경우가 뭐가 있을까?**

1. 책에서의 예시: 매개변수 타입만 사용하고 싶을 때
2. 래퍼 함수를 만들고 싶을 때

```ts
function log(...args: Parameters<typeof console.log>) {
  console.log('[LOG]', ...args);
}
```

`console.log`가 어떤 매개변수를 받는지 몰라도, `Parameters<typeof console.log>`로 그 타입을 그대로 가져와서 안전하게 위임할 수 있음.

3. 이벤트 핸들러나 콜백 타입을 재사용하고 싶을 때
