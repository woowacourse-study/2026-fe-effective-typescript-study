### 2026-07-21 · 아이템 4–5

#### 이번 스터디 핵심

타입스크립트는 객체의 이름이나 선언 관계보다 **객체가 가진 속성의 구조**를 기준으로 타입 호환성을 판단한다.
구조적 타이핑은 유연성과 테스트 편의성을 주지만, 타입이 닫혀 있다고 생각하면 예상하지 못한 동작을 만들 수 있다.
`any`는 타입 검사를 우회할 뿐 아니라 함수 계약, 언어 서비스, 리팩터링 안전성까지 함께 무너뜨린다.

#### 아이템 4: 구조적 타이핑에 익숙해지기

##### 구조적 타이핑이란?

자바스크립트에서는 객체의 실제 이름보다 객체가 어떤 속성과 메서드를 제공하는지가 중요하다. 이를 흔히 **덕 타이핑(duck typing)**이라고 한다.
타입스크립트는 자바스크립트의 동작 방식을 정적 타입 시스템으로 모델링하기 위해 **구조적 타이핑(structural typing)**을 사용한다.
어떤 값이 특정 타입이 요구하는 속성을 모두 가지고 있다면, 두 타입 사이에 명시적인 상속 관계가 없어도 호환되는 것으로 판단한다.

##### 예시

```ts
interface Vector2D {
  x: number;
  y: number;
}

function calculateLength(v: Vector2D) {
  return Math.sqrt(v.x ** 2 + v.y ** 2);
}

interface NamedVector {
  name: string;
  x: number;
  y: number;
}

const nv: NamedVector = {
  x: 3,
  y: 4,
  name: 'Pythagoras',
};

calculateLength(nv); // 5
```

`NamedVector`와 `Vector2D` 사이에는 명시적인 관계가 없다.
하지만 `NamedVector`는 `Vector2D`가 요구하는 `x`, `y` 속성을 모두 가지므로 `calculateLength`의 인자로 전달할 수 있다.

##### 타입은 열려 있다

함수의 매개변수 타입에 선언된 속성이 실제 인자가 가진 속성의 전부라고 단정하면 안 된다.
타입스크립트의 객체 타입은 “이 속성들만 존재한다”가 아니라 **“최소한 이 속성들은 존재한다”**라는 의미에 가깝다.
객체 리터럴을 변수에 바로 할당하거나 함수에 직접 전달할 때는 **잉여 속성 체크(excess property checking)**가 별도로 적용될 수 있다.
구조적 타이핑과 잉여 속성 체크를 같은 개념으로 혼동하지 않아야 한다.

##### 클래스와 테스트에서의 활용

클래스 인스턴스도 구조를 기준으로 호환성을 판단하며, 인스턴스 타입을 비교할 때 생성자 자체는 보통 고려 대상이 아니다.
테스트에서는 실제 클래스 전체를 생성하지 않고 필요한 최소한의 속성과 메서드만 가진 객체를 대역으로 사용할 수 있다.

##### 기억할 점

타입의 이름이 아니라 값의 구조가 호환성을 결정한다.
함수에는 선언된 속성 외에 추가 속성을 가진 값도 전달될 수 있다.
사람의 타입 이해와 타입 체커의 판단이 달라질 수 있음을 기억한다.

#### 아이템 5: `any` 타입 지양하기

타입스크립트의 타입 시스템은 **점진적**이며 **선택적**이다.
`any`는 기존 자바스크립트를 점진적으로 옮길 수 있게 하지만, 타입 시스템이 주는 이점을 포기하게 만든다.

##### 1. 타입 안전성을 없앤다

`any`는 어떤 타입에도 할당할 수 있고 어떤 타입의 값도 받을 수 있다.
타입 체커가 잡아야 할 오류가 실행 시점까지 숨어 있을 수 있다.

```ts
let value: any = 'hello';
const count: number = value; // 타입 검사를 통과한다.

count.toFixed(); // 실행 시점에 오류가 발생할 수 있다.
```

##### 2. 함수 시그니처를 무시한다

함수 시그니처는 호출자와 구현자 사이의 계약이다.
`any`를 사용하면 매개변수와 반환 타입에 담긴 계약을 우회할 수 있다.

##### 3. 언어 서비스를 약화한다

자동 완성, 속성 안내, 정의로 이동 같은 편집기 기능을 제대로 활용할 수 없다.
심벌 이름을 바꾸는 리팩터링에서도 변경 대상을 정확히 추적하기 어렵다.

##### 4. 리팩터링 오류를 감춘다

코드 양쪽의 계약이 함께 바뀌어야 할 때 한쪽이 `any`를 사용하면 불일치가 드러나지 않는다.
컴파일은 통과하지만 실행 중 오류가 발생하거나 잘못된 데이터가 전달될 가능성이 커진다.

##### `any` 대신 검토할 선택지

타입을 아직 모른다면 `unknown`을 사용하고, 사용 전에 타입을 좁힌다.
필요한 객체 구조를 인터페이스로 정의한다.
제네릭이나 유니온 타입으로 값 사이의 관계를 표현한다.
불가피하게 `any`를 사용한다면 범위를 최대한 좁히고 타입 경계에서 값을 검증한다.

#### 실전 체크리스트

객체 리터럴의 잉여 속성 체크와 구조적 타이핑을 혼동하고 있지 않은가?
테스트에서 실제 클래스 대신 필요한 구조만 가진 대역을 사용할 수 있는가?
`any`를 `unknown`, 구체적인 인터페이스, 제네릭 또는 유니온으로 바꿀 수 있는가?
`any`가 함수의 입력이나 반환값을 통해 다른 영역으로 퍼지고 있지 않은가?

#### 한 줄 정리

타입스크립트의 유연성은 구조적 타이핑에서 얻고, 그 유연성이 무너뜨릴 수 있는 안전성은 `any`를 피함으로써 지킨다.
